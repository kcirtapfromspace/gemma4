I.
Chapter One — The Baseline

A Jetson, half-asleep.

Twenty-five days from a hackathon deadline. The whole stack rests on an NVIDIA Jetson Orin NX 8 GB pinned, against the team's intent, to 7-watt mode — memory bandwidth EMC stuck at 2 133 MHz when 15 W mode would unlock 3 200 MHz.

A Pareto sweep on llama.cpp across five quantizations and three prompts named a winner — cliniq-gemma4-e2b-Q3_K_M.gguf — at 1.45 tok/s with perfect 1.00 extraction score. But that number was historical. Today the same kernel measured at 0.88 tok/s. The Jetson had drifted.

The cluster

Three Jetson Orin NX 8 GB boards on a Turing Pi 2 carrier, orchestrated by Talos Linux (immutable rootfs, k8s 1.35). One x86 control-plane node. Local container registry on the BMC — useful, occasionally unreliable.

The workload

Electronic Initial Case Reports — 200-token clinical narratives — extracted to minified FHIR-shaped JSON with SNOMED, LOINC, RxNorm codes. Real terminology. No invented codes allowed.

The 1.45 tok/s historical number could not be reproduced from artifacts on disk; commit 8da77e9's .so files had been overwritten mid-pivot. Team B's diagnosis filed at apps/mlc-llm/BASELINE.md.

II.
Chapter Two — Dead ends, surveyed

Five doors that did not open.

The agents fanned out. The first wave — five teams in parallel — took on power-mode unlocking, CUDA flag tuning, TensorRT-LLM feasibility, mobile feasibility, and speculative decoding. By midnight, four of them had filed honest no-go reports.

Each was a short essay on what infrastructure cannot bend to. The Talos kernel did not contain the nvpmodel driver. NVIDIA's TensorRT-LLM did not support Jetson sm_87 or Gemma 4 on any branch. The llama.cpp flag space had no free lunches at 7 W. Speculative decoding got +19 % on ngram-cache and nothing on a real draft model.

What they ruled out

Path A — 15 W power mode. Five independent proofs that the bottleneck was EMC bandwidth, not CPU. Image rebuild required.

Path B — TensorRT-LLM on Jetson. Four stacked blockers across runtime, model arch, container tags, and LoRA format. Estimated effort to override: 10–21 days.

Path C — CUDA flag tuning. Zero net wins under the 7 W regime; one -27 % regression from KV-cache int8 quant.

Path D — Speculative decoding. Target model already too small. No materially smaller draft model with the same tokenizer existed.

The Pivot
"

The clinic uses a phone, not a cluster.

Mid-session course-correction · ~04-23 17:30 local

One sentence reframed the entire problem. If the deployment target is a clinician's phone in a remote dead zone — not a Jetson on a workbench — then the optimization isn't tok/s on Talos. It's tok/s on Apple silicon. The agent swarm pivoted within thirty minutes.

III.
Chapter Three — The mobile stack

A six-step pipeline, stitched live.

Once the target became a phone, eleven agents reconverged around a single critical path. Train on Kaggle. Re-merge to fp16 safetensors locally. Convert to .litertlm via Google's pre-release LiteRT-LM toolchain — and, in parallel, to GGUF for llama.cpp. Build a Swift wrapper around LiteRT-LM that Google hadn't shipped yet. Sideload onto the iPhone 17 Pro simulator. Drive it from a SwiftUI clinician interface that doesn't show JSON.

V.
Chapter Five — Walls, scaled

Eight stack traces, eight workarounds.

Every shipped pipeline has a graveyard of error messages that came before. These eight got close to canceling the demo.

OSError: /usr/lib/aarch64-linux-gnu/nvidia/libnvrm_gpu.so: file too short

Jetson NVIDIA libs evaporated after a power cycle.

The dustynv image ships zero-byte stub .so's that the runtime is supposed to overlay. Talos doesn't. Pulled real libraries off the host with talosctl cp, tar-piped them through kubectl exec into the pod.
Resolved · 04-23 16:20
ImportError: cannot import name 'ConstantLengthDataset' from 'trl.trainer.utils'

The Kaggle TRL API moved while we slept.

Three retries, three different missing symbols, three different TRL submodules. Final fix: bypass unsloth's chat-template helper, embed the gemma-4 template inline, replace DataCollatorForCompletionOnlyLM with a 30-line subclass.
Resolved · 04-24 00:43
ValueError: Target module Gemma4ClippableLinear is not supported

PEFT did not recognize Gemma 4's clipped projections.

Walk named_modules, replace each ClippableLinear with its inner Linear in place — 232 layers unwrapped — before attaching the adapter. Same trick the training kernel used.
Resolved · 04-24 09:46
NotImplementedError: Quant method is not yet supported: 'bitsandbytes'

The Kaggle merge was 4-bit nf4, not fp16.

llama.cpp's converter rejects bnb safetensors. Re-merged locally from the cached fp16 base + the saved PEFT adapter in 20 seconds. Saved 9.4 GB clean fp16 ready for GGUF.
Resolved · 04-24 09:48
AttributeError: type object 'MODEL_ARCH' has no attribute 'GEMMA4'

The packaged gguf module trailed llama.cpp HEAD.

Sparse-checkout of llama.cpp HEAD's gguf-py directory. PYTHONPATH=/tmp/llama-cpp-head/gguf-py on the converter. Build resumed in seconds.
Resolved · 04-24 09:49
No space left on device · 32 GB free of 926 GB on data volume

The conversion pipeline wanted 60 GB of headroom.

Surveyed every artifact, classified by regenerable / not. Moved 14.9 GB of non-winner GGUFs, 9 GB of stale MLC build output, and a 3.5 GB Rust target/ to the NAS. Wrote NAS_OFFLOAD.md restore manifest.
Resolved · 04-24 01:04
JSON.parseError · Expecting ',' delimiter at column 268

The fine-tune emits valid keys with broken quoting.

Auto-balance unclosed { / [ after max_tokens cutoff. Promote arrays nested inside patient. Fall back to a regex extractor when JSONSerialization still rejects. Now tolerates four classes of LLM JSON drift.
Resolved · 04-24 09:58
EngineFactory::CreateDefault() → NotFound

Apple's linker dead-stripped the engine registration.

LiteRT-LM uses file-scope static initializers to register its engine variants. Without -Xlinker -all_load on the SwiftPM binary target, the symbols don't survive. Once added, the first testEngineLoadsModel turned green.
Resolved · 04-23 23:40

Coda — The shipped surface

Four entities, three taps, no internet.

On a real iPhone, this same flow runs on Metal-accelerated GPU. Projected on Google's published LiteRT-LM benchmarks for the same model: 52–56 tok/s on iPhone 17 Pro, 52 tok/s on Samsung S26 Ultra, ~10 s total for a full clinical extraction.

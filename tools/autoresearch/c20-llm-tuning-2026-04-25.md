### v27 conversion blocker — diagnosis

The kernel produced two artifacts but neither is locally convertible to GGUF:

- `cliniq-compact-merged/` — model with `quantization_config: bnb_4bit` in
  config.json, weights stored as uint8 with `absmax` / `quant_map` aux
  tensors (not standard fp16). `convert_hf_to_gguf.py` rejects this:
  `NotImplementedError: Quant method is not yet supported: 'bitsandbytes'`.
- `cliniq-compact-lora/` — LoRA adapter targeting
  `Gemma4ClippableLinear` modules. PEFT 0.19.0 doesn't have a LoRA
  registration for this custom module type:
  `ValueError: Target module Gemma4ClippableLinear(...)`.

**Two fix paths, both kernel-side:**

1. **Re-run Kaggle kernel** with
   `model.save_pretrained_merged(out, tokenizer, save_method="merged_16bit")`
   — produces a standard fp16 HF dir that `convert_hf_to_gguf.py` reads.
2. **Install Unsloth in scripts/.venv** + dequantize the bnb safetensors via
   `bitsandbytes.functional.dequantize_4bit` per-tensor, then save fp16.
   More invasive locally; cleaner to fix at source.

Either way, pre-staged: `scripts/v27_convert_and_bench.sh` runs the verdict
pipeline (bench against base, keep/discard) once a clean fp16 merged dir
lands.

**Honest next-action:** push the kernel-side fix to a follow-up sprint.
The c20 sprint's conclusion ("model at quality ceiling on this bench")
makes v27 strictly evaluative — F1 = 1.000 already holds without v27,
and the most likely outcome is "v27 either matches base or breaks
tool-calling," neither of which changes the demo claim.

### Candidate D result — short-circuit gate refinement (ios-eng, 2026-04-25 PM)

**Chosen criterion (hybrid, "Option A++"):** the gate
`ParsedExtraction.shortCircuitsLLM` fires when **either**
(a) at least one match came from an explicit-assertion tier — inline
parenthesized `(SNOMED 12345)` or CDA `<code .../>` — **or**
(b) deterministic populated codes in **>=2 of the 3 buckets**
(conditions / loincs / rxnorms). Otherwise (single-bucket
*lookup-only* result) we fall through to the fast-path / agent so the
LLM can backfill the missing axis.

**Rationale.** Pure ">=2 buckets" (Option A) over-triggered the agent on
`adv2_negation_ruled_out` (only 1 expected SNOMED, surfaced via inline
`(SNOMED 31541009)`) — letting the agent run there has no upside and
opens an FP risk surface. Adding the explicit-assertion bypass keeps
all author-named codes on the deterministic path while only widening
the gate for the lookup-tier-only cases that motivated this work.
The inline/CDA tiers carry confidence floors of 0.99 vs lookup's 0.85
(see `EicrPreparser.tierConfidence*`), so this stratification is
already present in the data model.

**Diagnostic correction.** The c20 "Live finding" subsection above
attributed the 5 misses to LOINC. Re-running deterministic against
each case in this worktree showed the misses on adv3_rmsf_rag and
adv3_valley_fever_rag are actually the **SNOMED** axis (RMSF SNOMED
186788009 and Coccidioides 37436014 are RAG-only entries, not in the
lookup table). Deterministic recovers the RxNorm via the antibiotic
alias. The fix and recovery still apply — just the missed axis is
RxNorm-not-LOINC for those two cases.

**Window-size deviation (flagged for review).** The original spec
called for a strict 30-char post-hoc window. The implementation
uses 80 chars (clause-bounded) because the actual adv5 dengue
narrative has ~33 chars between "Zika" and "came back negative":

> "Zika virus IgM antibody serology came back negative"

A 30-char post-hoc window only reaches the 'c' of 'came'. Widening
to 80 chars (still bounded by the next `.`, `;`, `,`, or newline)
catches the trigger without enabling cross-sentence false positives.
Because the post-hoc trigger phrases are multi-word and
discriminative (e.g., "came back negative"), over-firing on this
wider window is extremely unlikely on clinical prose. The standard
60/30-char windows for `_NEG_TRIGGERS` are unchanged.

adv2, adv3, adv4 deterministic sets unchanged: 0 false positives
across all of them. The new post-hoc and pre-hoc triggers fired
on exactly the two target adv5 cases — no spillover.

**Cases the c20 doc should track for follow-up.**

- The 80-char post-hoc window is wider than the spec's
  conservative 30 chars. If a future adv6 case introduces
  cross-clause noise that exploits this — e.g., a sentence where
  "came back negative" sits in a parenthetical 70 chars after a
  *non-negated* concept — the fix would be to terminate the
  post-hoc scan more aggressively (e.g., add `(`, `)` to the
  post-hoc terminator set) or adopt a per-clause syntactic parse.
  No such case exists in combined-45 today.
- "Differential included" was added to the pre-hoc trigger set as
  a conservative extension of the existing `differential
  diagnosis|dx` rule. The morphology variant "Differential
  considered" / "Differential workup" is NOT yet covered. Worth
  adding pre-emptively if it surfaces in a future bench.
- `not\s+invoking` is narrow. Real-world clinician phrasing also
  uses "not entertaining", "not pursuing", "not chasing", "not
  considering". These would be sensible additions if a future bench
  sees them; held back here to keep the fix conservative.
- Fast-path top-1 limitation: `RagSearch.fastPathHit` only inspects
  the top RAG hit. On `adv5_ruled_in_measles_ruled_out_rubella`,
  Rubella scores higher than Measles (1.365 vs 1.122). With the
  fix, Rubella is correctly suppressed → fast-path declines → the
  agent path recovers Measles in 24 s. A future enhancement could
  let the fast-path try the next-highest hit when top is negated,
  saving the agent latency. Out of scope for this fix.

**New failure modes surfaced (5 of 9):**

1. **NegEx terminator gap on commas**
   (`adv6_neg_enumeration_history_seizures_no_stroke`).
   `_NEG_TERMINATORS = (?:\bbut\b|\bhowever\b|\balthough\b|\bexcept\b|\.\s|;|\n|<)` does not include `,`.
   On comma-separated PMH lists like
   "no history of stroke, history of HIV infection" the trigger
   leaks past the comma and suppresses the next clause's positive
   diagnosis. Fix: add `,` (or `,\s*`) to the terminator regex.
   Risk: comma-bounded NegEx is more aggressive than period-bounded;
   could under-fire on legitimate "no evidence of X, Y, or Z"
   enumerations. Worth a regression sweep before shipping.

2. **NegEx trigger gap on "do not have" / "not eligible for"**
   (`adv6_quoted_negative_covid`).
   The current `_NEG_TRIGGERS` has `not\s+detected`, `not\s+positive\s+for`,
   `not\s+suspected`, `not\s+invoking` but NOT the broader negation
   forms `do(?:es)?\s+not\s+have`, `not\s+eligible\s+for`,
   `no\s+current\s+evidence\s+of`. Fix: extend the trigger set.
   Conservative additions: `do(?:es)?\s+not\s+have`, `did\s+not\s+have`,
   `not\s+eligible\s+for`. The "not entertaining" / "not pursuing"
   variants from the c19 follow-up note still apply.

3. **Inline regex emits any 6-9 digit number after "SNOMED"**
   (`adv6_code_injection_bait`). `SNOMED_RE` is lexical-only with
   confidence=0.99. There is no allow-list check (e.g., does
   the code exist in the curated SNOMED universe?), no quoted-context
   scope, and no NegEx applied to inline tier. A clinician (or a
   patient quoting the internet) writing "(SNOMED 99999999)" in
   prose will be extracted at confidence 0.99. Fix options:
   (a) post-filter inline matches against a curated SNOMED allow-list,
   (b) apply `_is_negated` to inline matches too,
   (c) detect quote context (single-quoted, double-quoted, or
   inside known "internet rumor" framing) and demote.
   Recommendation: (a) for hackathon — cheap, deterministic, stops
   the worst case. (b) is structurally cleaner but doesn't help when
   the bait is just citing without negating.

4. **Unicode whitespace breaks alias regex**
   (`adv6_unicode_typography_em_dash_smart_quotes`).
   `re.escape("Bordetella pertussis DNA NAA")` puts a literal U+0020
   space between the words; if the input has U+00A0 (non-breaking
   space) — which is common in pasted clinical text from EHR PDFs —
   the pattern misses. Fix: pre-normalize input with
   `unicodedata.normalize("NFKC", text).replace(" ", " ")`
   (and similar for thin-space, en-space) before matching, or
   compile aliases with `\s+` between tokens instead of literal spaces.
   Cheap, no risk.

5. **Deterministic FPs poison the fast-path gate**
   (`adv6_long_form_admission_note`).
   `try_fast_path` skips the RAG path if `det_codes` is non-empty.
   When det emits FPs (varicella from "no varicella series", CBC
   from "CBC: WBC..."), the gate concludes Tier 1 succeeded and
   never asks RAG. Measles (which IS in `reportable_conditions.json`)
   is therefore never recovered. Fix options:
   (a) gate on det confidence — only short-circuit if any match
   has tier in {inline, cda} (the Cand D rule already does this
   for `_det_short_circuits_llm`, but `try_fast_path` does not),
   (b) tighten NegEx so "no varicella series" suppresses the
   varicella alias (extend `_NEG_TRIGGERS` with `^\s*-\s*no\s+`
   or include the specific "no X series/coverage" pattern),
   (c) treat "CBC" alias as ambiguous and require a co-located
   "test"/"panel"/"value" qualifier.
   Recommendation: (a) is the cleanest; the Cand D logic already
   knows which tiers are explicit assertions vs lookup aliases.
   Mirror that into `try_fast_path` so lookup-only FPs don't block
   the RAG fallback.

6. **(Bonus) RAG embedding can match 3-letter alt_names cross-domain**
   (`adv6_implicit_syndrome_stevens_johnson`).
   Fast-path emitted CRE SNOMED 47523006 at score 1.0 for an SJS
   narrative because "CRE" is one of CRE's alt_names and the
   embedding/scoring rewarded a 3-letter token match somewhere.
   Recommendation: cap fast-path on alt_name matches that are
   shorter than 4 characters unless a co-located explicit qualifier
   is found ("CRE infection", "carbapenem-resistant"). Or simply
   drop the standalone "CRE" alt_name from the curated DB and
   require the longer phrase. (The negative case the user authored
   — SJS, which we don't curate — turned into a *positive* on a
   completely unrelated condition. This is a calibration issue
   distinct from the recall miss.)
